# 备件管理系统 · 上线方案全面对比分析

> 本文档记录系统上线部署的完整方案分析、技术选型推演与最终决策依据。
> 涵盖从初始 4 候选方案 → 深度评估 → 方案修正 → 最终对比的全过程。

---

## 一、背景：核心问题

当前系统分为两个独立技术栈，它们的数据库天然隔离：

```
── 当前开发架构 ──────────────────────────────────────

  PC 管理后台                    微信小程序
  Vue3 + Element Plus           原生 WXML/JS
  + ECharts                     5 个页面
       │                            │
  Express API (localhost:5501)   云函数 ×6 (微信云开发)
  32 个接口 · 8 个 handler          │
       │                            │
  MongoDB (localhost:27017)      云数据库 (微信云开发)

  ⚠️ 两套后端 + 两套数据库 → 数据无法互通
```

**上线核心挑战**：如何让 PC 管理后台和微信小程序共享同一数据源，实现实时数据互通。

---

## 二、初始候选方案（A / B / C / D）

### 2.1 方案总览

| # | 方案 | 原理 | 优点 | 缺点 |
|:-:|------|------|------|------|
| A | PC 后台直连微信云数据库 | `@cloudbase/node-sdk` HTTP API 读写云开发数据库 | 架构简单，无额外服务器 | 聚合能力受限，API 语法需全面改写 |
| B | 云托管容器 + 独立 MongoDB | Express Docker 化部署到微信云托管，内网连接独立 MongoDB | Express 代码零改动，MongoDB 全功能 | 需额外购买 MongoDB 实例 |
| C | 中间件双向同步 | MongoDB Change Stream → 云数据库实时同步 | 保留现有架构不变 | 架构最复杂，数据冲突难处理 |
| D | 云函数作 API 网关 | PC 后台也通过 HTTP 调用云函数 | API 逻辑统一 | 延迟高，本地开发体验差 |

### 2.2 快速淘汰：方案 C 和 D

**方案 C（中间件双向同步）** — 淘汰

- MongoDB Change Stream 要求副本集架构，本地开发环境需配置 oplog
- 双向同步存在数据冲突风险（PC 端和小程序同时修改同一条记录）
- 需要额外开发同步中间件，引入额外故障点
- 架构复杂度最高，小团队维护成本过大

**方案 D（云函数作 API 网关）** — 淘汰

- 每次 HTTP 调用云函数额外增加 100-300ms 延迟（冷启动更长）
- PC 端数据分析页面需要并行请求 3-7 个接口，延迟叠加体验极差
- 本地开发时也需要联网调用云函数，离线无法开发
- 32 个接口全部走云函数网关，调试链路过长

**结论**：仅保留方案 A 和方案 B 进入深度评估。

## 三、方案 A 深度分析：PC 后台直连微信云数据库

### 3.1 方案架构

```
  PC 浏览器                    微信小程序
      │                          │
  Vue3 (部署到服务器/CDN)     原生 WXML/JS
      │                          │
  Express (部署到服务器)       云函数 ×6
      │                          │
      │  @cloudbase/node-sdk     │  wx-server-sdk
      └────── 微信云数据库 ───────┘   ← 数据互通 ✅
```

### 3.2 聚合能力局限性（关键问题）

经微信官方文档验证，云开发数据库与 MongoDB 原生驱动的聚合能力存在显著差异：

| 聚合操作 | MongoDB 原生 | 微信云数据库 | 当前代码使用位置 | 未来扩展需求 |
|----------|:------:|:------:|------|------|
| `$lookup` (跨集合 JOIN) | ✅ | ✅ | analytics.js 安全库存预警 | 多表关联报表 |
| `$switch` | ✅ | ✅ | analytics.js 库龄分桶 | — |
| `$dateToString` | ✅ | ✅ | analytics.js 月度趋势 | — |
| `$dateDiff` | ✅ | ❌ | analytics.js 库龄计算 (2处) | 时间差分析 |
| `$facet` (并行多管道) | ✅ | ❌ | 当前未用 | 仪表盘多维度一次查询 |
| `$merge` / `$out` (聚合写入) | ✅ | ❌ | 当前未用 | 报表预计算/ETL |
| `$graphLookup` (递归查找) | ✅ | ❌ | 当前未用 | 多级审批/BOM 树 |
| `$unionWith` (多集合合并) | ✅ | ❌ | 当前未用 | 跨表综合报表 |

**直接影响**：
- `analytics.js` 中 2 处 `$dateDiff` 调用需要改写为 `$subtract` + 手动毫秒换算
- 未来如需 `$facet` 做仪表盘多维度聚合，方案 A 完全无法支持

### 3.3 隐性成本

| 问题 | 说明 |
|------|------|
| **API 语法不兼容** | `@cloudbase/node-sdk` 使用链式调用风格 (`db.collection('x').where({}).get()`)，与 MongoDB 原生驱动语法 (`db.collection('x').find({})`) **完全不同**。8 个 handler 文件的所有数据库操作都需要改写或维护双版本 |
| **外网延迟** | PC 后台通过外网 HTTP 调用云数据库，每次请求额外 20-80ms 延迟。数据分析页面并行 3-7 个请求，体验明显下降 |
| **双版本维护** | 若保持本地开发用 MongoDB 原生驱动，则需维护两套数据库操作代码（本地/线上），增加 bug 风险 |

### 3.4 方案 A 评价

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 架构简洁性 | ⭐⭐⭐ | 不需要额外 MongoDB 实例 |
| 代码改动量 | ⭐ | 8 个 handler 文件全部需要改写数据库调用 |
| MongoDB 功能 | ⭐ | 聚合能力受限，`$dateDiff`/`$facet` 不可用 |
| 未来扩展性 | ⭐ | 数据分析需求受限于云数据库能力天花板 |
| 开发体验 | ⭐⭐ | 本地/线上两套语法，调试痛苦 |

## 四、方案 B 深度分析与修正

### 4.1 关键事实纠正

经查证微信官方文档，最初对方案 B 的理解存在重大误区：

| 事实 | 说明 | 来源 |
|------|------|------|
| 云托管自带的是 **MySQL**，不是 MongoDB | 不能直接跑现有 Express + MongoDB 代码 | 微信云托管 FAQ |
| 云托管**不能直连**微信云开发数据库 | 两者是独立体系，无法通过内网 MongoDB 协议互通 | 微信官方文档 |
| 唯一桥接方式: Node.js SDK | 走 SDK API = 与方案 A 同样的聚合限制 | 微信云托管 FAQ |
| 要用原生 MongoDB | 必须额外购买**腾讯云 MongoDB 实例**，同 VPC 内网连接 | 微信云托管 FAQ |

**因此方案 B 分裂为两条子路线**：

- **B-1**：云托管 + 腾讯云 MongoDB — Express 代码零改动，MongoDB 全功能
- **B-2**：云托管 + Node.js SDK → 云开发数据库 — 与方案 A 同样的聚合限制（**淘汰**）

以下仅分析 B-1。

### 4.2 方案 B-1 架构：云托管 + 腾讯云 MongoDB

```
┌──────────────────── 微信云开发环境 ────────────────────┐
│                                                       │
│  ① 微信云托管 (Docker)         ② 云函数 ×6             │
│     Express :5501                (小程序专用, 保持不变)  │
│     JWT 认证                     微信 openId 认证       │
│         │                            │                 │
│         └──── 同 VPC 内网 ───────────┘                 │
│                      │                                 │
│              ┌───────▼────────┐                        │
│              │ 腾讯云 MongoDB  │                        │
│              │ (独立实例)      │                        │
│              │ 唯一数据源      │                        │
│              └────────────────┘                        │
│                                                       │
│  ③ 静态托管 / CDN                                      │
```

## 五、方案 A vs B-1 对比

| 维度 | A：直连云数据库 | B-1：云托管 + 腾讯云 MongoDB |
|------|:---:|:---:|
| **数据互通** | ✅ | ✅ |
| **Express 代码改动** | ❌ 8 个 handler 全部改写 | ✅ 仅改 db.js 1 行 |
| **MongoDB 聚合功能** | ❌ 受限（无 `$dateDiff`/`$facet`） | ✅ 完整 |
| **本地开发体验** | ⚠️ 需维护双版本语法 | ✅ 完全不变 |
| **性能** | ⚠️ 外网延迟 20-80ms/请求 | ✅ VPC 内网 <1ms |
| **额外购买** | 无 | 腾讯云 MongoDB ¥200-545/月 |
| **月度成本** | ¥25-75 | ¥225-620 |
| **架构统一性** | ⚠️ 两套后端+API 语法不同 | ⚠️ 两套后端+共享数据库 |
| **未来扩展** | ❌ 受云数据库能力天花板限制 | ✅ 无限制 |
| **运维复杂度** | 低 | 中（Docker + VPC + MongoDB 实例） |

**结论**：
- 方案 A 成本低但技术限制大，handler 改写工作量巨大，且限制未来扩展
- 方案 B-1 成本高但技术上更合理，代码改动最小，MongoDB 全功能可用
- **如果选择保留微信小程序，推荐 B-1**

## 六、跳出微信：方案 B-2（小程序调外部 API）

### 6.1 方案思路

保留微信小程序，但不使用微信云开发，小程序直接调用外部自建 Express API。

```
  PC 浏览器                    微信小程序
      │                          │
  Vue3                       原生 WXML/JS
      │                          │
      └──── Express API ─────────┘
                  │
             MongoDB (自建)

  ✅ 数据互通（共用同一个 Express 后端）
```

### 6.2 实现步骤

| 阶段 | 工作项 | 说明 |
|:----:|--------|------|
| 1 | 购买云服务器 | 轻量应用服务器，运行 Express + MongoDB |
| 2 | 域名备案 | ICP 备案（7-25 天），微信小程序要求服务器域名必须备案 |
| 3 | SSL 证书 | 小程序要求所有请求必须 HTTPS |
| 4 | 服务器部署 | Express + MongoDB + Nginx 反向代理 |
| 5 | 小程序改造 | `wx.request()` 替代 `wx.cloud.callFunction()`，6 个页面全部改造 |
| 6 | 微信后台配置 | 配置服务器域名白名单 |
| 7 | 认证改造 | 小程序端实现 `wx.login()` → 自建后端换取 JWT |

### 6.3 痛点

- 需要自建服务器和运维
- 域名 ICP 备案周期长
- 小程序所有页面的数据调用方式需要重写（`wx.cloud.callFunction` → `wx.request`）
- 6 个云函数的代码投入成为沉没成本

### 6.4 方案 B-2 评价

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 架构简洁性 | ⭐⭐⭐⭐ | 一套后端（Express），两套前端 |
| 代码改动量 | ⭐⭐⭐ | 小程序 6 个页面调用方式重写 |
| MongoDB 功能 | ⭐⭐⭐⭐⭐ | 自建 MongoDB，完整功能 |
| 月度成本 | ⭐⭐⭐⭐ | ¥45-105/月 |
| 微信能力保留 | ✅ | 扫码、订阅消息均保留 |

## 七、方案 C（纯 Web）：完全替代微信小程序

### 7.1 方案概述

完全放弃微信小程序，使用响应式 Web 应用覆盖 PC + 手机端。

```
── 纯 Web 方案 ─────────────────────────────────────

  PC 浏览器                  手机浏览器
      │                          │
      └──── Vue3 (响应式布局) ────┘
                   │
              Express API (唯一后端)
                   │
              MongoDB (唯一数据库)

  ✅ 一套前端 + 一套后端 + 一套数据库 → 天然互通
```

### 7.2 功能替代可行性

| 功能 | 微信小程序实现 | 纯 Web 替代方案 | 可行性 |
|------|---------------|----------------|:------:|
| 用户登录 | 微信 openId 静默登录 | 用户名+密码（已有 JWT 体系） | ✅ 现成 |
| 扫码查询/入库 | `wx.scanCode()` 原生调用 | HTML5 `getUserMedia` + jsQR 库 | ⚠️ 中等 |
| KPI 仪表盘 | 自写布局 | PC 端已有 ECharts，移动端复用 | ✅ 简单 |
| 库存列表/搜索 | 自写列表 + 下拉刷新 | Element Plus 表格 / 移动端列表 | ✅ 简单 |
| 申请出库 | 自写表单 | PC 端已有完整表单 | ✅ 现成 |
| 审批管理 | 自写弹窗 | PC 端已有审批弹窗 | ✅ 现成 |
| 消息通知 | 微信订阅消息（3 个场景） | 邮件 / Web Push / 企业微信 Webhook | ⚠️ 替代方案 |
| 离线使用 | 小程序离线缓存 | PWA Service Worker | ⚠️ 需额外开发 |
| 手机桌面图标 | "添加到桌面" | PWA "添加到主屏幕" | ✅ 简单 |
| 团队分发 | 微信内分享/扫码 | 发链接/二维码 | ✅ 更简单 |

### 7.4 纯 Web 方案的优势

| 维度 | 说明 |
|------|------|
| **架构统一** | 一套 Vue3 前端 + 一套 Express 后端 + 一套 MongoDB，彻底消除互通问题 |
| **代码复用最大化** | PC 端已有的全部 32 个 API、8 个 handler、所有页面组件都直接复用，不需要维护两套代码 |
| **MongoDB 全功能** | `$lookup` / `$facet` / `$dateDiff` 全部可用，数据分析扩展无限制 |
| **成本最低** | 一台 ¥40-100/月的轻量服务器搞定全部，不需要腾讯云 MongoDB（¥200-545/月） |
| **开发效率** | 只维护一个项目，前后端本地调试体验统一 |
| **不受平台约束** | 无需微信审核、无需微信开发者工具、不受微信 API 限制 |
| **跨平台** | iOS / Android / PC / 平板，任何有浏览器的设备都能用 |
| **团队分发更简单** | 发个链接即可，不需要微信扫码、不需要关注、不需要体验版名额 |

### 7.5 纯 Web 方案的劣势

| 维度 | 说明 | 严重程度 |
|------|------|:--------:|
| **扫码体验差一些** | HTML5 Camera API 需要用户授权，识别速度略慢于微信原生 `scanCode` | ⚠️ 中 |
| **消息通知弱化** | 失去微信服务通知的强触达能力，替代方案（邮件/企业微信）触达率低一些 | ⚠️ 中 |
| **无微信生态入口** | 不能在微信内分享小程序卡片，需要发链接 | 低 |
| **需自运维服务器** | 需要自己维护服务器安全、备份、更新 | ⚠️ 中 |
| **已有小程序代码废弃** | `miniprogram/` 下 5 个页面 + 6 个云函数 + 订阅消息模块的开发投入沉没 | 高 |
| **手机浏览器体验** | 不如原生小程序流畅（无预加载、下拉刷新需自己实现） | 低 |

## 十、最终结论与推荐

### 按场景推荐

**场景 1：预算敏感 + 小团队 + 看重长期维护**
> **推荐方案 C（纯 Web）**
> - 月度成本 ¥45-105，架构最简
> - PC 端代码几乎全部复用，只需新建移动端响应式视图
> - 微信小程序 ~3800 行代码沉没，但长期收益远大于短期损失

**场景 2：需要保留微信小程序 + 控制成本**
> **推荐方案 B-2（小程序调外部 API）**
> - 月度成本 ¥45-105，保留微信扫码和订阅消息
> - 需要重写小程序调用方式，但后端完全不变

**场景 3：需要保留微信小程序 + 不计成本**
> **推荐方案 B-1（云托管 + 腾讯云 MongoDB）**
> - 月度成本 ¥225-620，代码改动最小
> - MongoDB 全功能，VPC 内网低延迟

**不推荐**：
- **方案 A**：虽然成本最低，但 8 个 handler 全部改写数据库调用的工作量巨大，且聚合能力受限限制未来扩展
- **方案 C/D**：架构复杂度过高（初始淘汰轮）

### 综合推荐排序

```
│  🥈 方案 B-2 (外部 API) — 保留微信 · 合理成本 · 架构统一        │
```
