# 备件管理系统 · 技术方案对比分析

> 本文档记录系统上线方案的完整分析与对比，为技术选型提供决策依据。

---

## 一、微信小程序上线方案分析（方案 B-1 vs B-2）

### 1.1 核心问题：PC 后台与微信小程序的数据互通

当前系统分为两个独立技术栈，它们的数据库天然隔离：

```
── 当前架构 ──────────────────────────────────────────

  PC 管理后台                    微信小程序
  Vue3 + Element Plus           原生 WXML/JS
       │                            │
  Express API (localhost:5501)   云函数 ×6 (微信云开发)
       │                            │
  MongoDB (localhost:27017)      云数据库 (微信云开发)

  ⚠️ 两套后端 + 两套数据库 → 数据互通是核心挑战
```

### 1.2 方案 B-1：PC 端迁入微信云

**思路**：将 PC 后台的 Express 应用容器化，部署到微信云托管，连接同一个微信云数据库。

```
  PC 浏览器                    微信小程序
      │                          │
  Vue3 (静态托管)            原生 WXML/JS
      │                          │
  Express (云托管容器)         云函数 ×6
      │                          │
      └────── 微信云数据库 ───────┘    ← 数据互通 ✅
```

**实现步骤**：

| 阶段 | 工作项 | 说明 |
|:----:|--------|------|
| 1 | Dockerfile 编写 | Express 应用容器化 |
| 2 | 云托管配置 | 微信云开发 → 云托管 → 部署容器 |
| 3 | VPC 网络打通 | 云托管容器通过内网访问云数据库 |
| 4 | 云函数改造 | 6 个云函数从 `wx-server-sdk` 改为连接云数据库 |
| 5 | 前端部署 | Vue3 构建产物部署到微信云开发静态托管 |
| 6 | 域名配置 | 自定义域名 + SSL → 指向云托管 |

**痛点**：
- 云数据库查询语法受限（不支持 `$lookup`、`$facet`、`$dateDiff` 等聚合管道）
- 需要额外购买腾讯云 MongoDB 实例（¥200-545/月）才能使用标准 MongoDB 语法
- VPC 网络配置复杂，调试困难
- 6 个云函数需要逐一改造连接方式

### 1.3 方案 B-2：小程序端迁出微信云

**思路**：小程序不使用云开发，改为直接调用外部 Express API。

```
  PC 浏览器                    微信小程序
      │                          │
  Vue3                       原生 WXML/JS
      │                          │
      └──── Express API ─────────┘
                  │
             MongoDB (自建)

  ← 数据互通 ✅（共用同一个 Express 后端）
```

**实现步骤**：

| 阶段 | 工作项 | 说明 |
|:----:|--------|------|
| 1 | 购买云服务器 | 轻量应用服务器，运行 Express + MongoDB |
| 2 | 域名备案 | ICP 备案（7-25 天），微信小程序要求服务器域名必须备案 |
| 3 | SSL 证书 | 小程序要求所有请求必须 HTTPS |
| 4 | 服务器部署 | Express + MongoDB + Nginx 反向代理 |
| 5 | 小程序改造 | `wx.request()` 替代 `wx.cloud.callFunction()`，6 个页面全部改造 |
| 6 | 微信后台配置 | 配置服务器域名白名单 |
| 7 | 认证改造 | 小程序端实现 `wx.login()` → 自建后端换取 JWT |

**痛点**：
- 需要自建服务器和运维
- 域名 ICP 备案周期长
- 小程序所有页面的数据调用方式需要重写

### 1.4 方案 B-1 vs B-2 对比表

| 维度 | B-1：PC 迁入微信云 | B-2：小程序迁出微信云 |
|------|:--:|:--:|
| **数据互通** | ✅ 云数据库统一 | ✅ 自建 MongoDB 统一 |
| **MongoDB 功能** | ⚠️ 云数据库受限 / 需额外购买 | ✅ 完整功能 |
| **月度成本** | ¥225-620 | ¥45-105 |
| **域名备案** | 不需要 | 需要（7-25 天） |
| **运维负担** | 低（云平台托管） | 中（自建服务器） |
| **开发改造量** | 大（容器化 + VPC + 云函数改造） | 中（小程序调用方式重写） |
| **架构一致性** | ⚠️ 两套后端（Express + 云函数） | ✅ 一套后端（Express） |
| **平台依赖** | 高（深度绑定微信云生态） | 低（仅用微信登录能力） |
| **调试体验** | ⚠️ 云端调试较困难 | ✅ 本地调试一致 |

### 1.5 推荐结论

**推荐方案 B-2**，理由：
1. 月度成本降低 80%（¥45-105 vs ¥225-620）
2. 架构统一为单一后端，长期维护简单
3. MongoDB 全功能可用，数据分析无限制
4. 虽然前期需要备案等待，但一劳永逸

## 二、纯 Web 方案分析（替代微信小程序）

### 2.1 方案概述

完全放弃微信小程序，使用响应式 Web 应用覆盖 PC + 手机端。

```
── 纯 Web 方案 ─────────────────────────────────────

  PC 浏览器                  手机浏览器
      │                          │
      └──── Vue3 (响应式布局) ────┘
                   │
              Express API (唯一后端)
                   │
              MongoDB (唯一数据库)

  ✅ 一套前端 + 一套后端 + 一套数据库 → 天然互通
```

### 2.2 功能替代可行性

| 功能 | 微信小程序实现 | 纯 Web 替代方案 | 可行性 |
|------|---------------|----------------|:------:|
| 用户登录 | 微信 openId 静默登录 | 用户名+密码（已有 JWT 体系） | ✅ 现成 |
| 扫码查询/入库 | `wx.scanCode()` 原生调用 | HTML5 `getUserMedia` + jsQR 库 | ⚠️ 中等 |
| KPI 仪表盘 | 自写布局 | PC 端已有 ECharts，移动端复用 | ✅ 简单 |
| 库存列表/搜索 | 自写列表 + 下拉刷新 | Element Plus 表格 / 移动端列表 | ✅ 简单 |
| 申请出库 | 自写表单 | PC 端已有完整表单 | ✅ 现成 |
| 审批管理 | 自写弹窗 | PC 端已有审批弹窗 | ✅ 现成 |
| 消息通知 | 微信订阅消息（3 个场景） | 邮件 / Web Push / 企业微信 Webhook | ⚠️ 替代方案 |
| 离线使用 | 小程序离线缓存 | PWA Service Worker | ⚠️ 需额外开发 |
| 手机桌面图标 | "添加到桌面" | PWA "添加到主屏幕" | ✅ 简单 |
| 团队分发 | 微信内分享/扫码 | 发链接/二维码 | ✅ 更简单 |

### 2.3 实施准备清单

#### 必须项

| # | 项目 | 说明 | 费用 |
|:-:|------|------|------|
| 1 | **云服务器 (VPS)** | 运行 Express + MongoDB | ¥40-100/月 |
| 2 | **已备案域名** | 浏览器访问必须 | ¥50-70/年 + 备案 7-25 天 |
| 3 | **SSL 证书** | HTTPS（摄像头扫码必须 HTTPS） | 免费（Let's Encrypt） |
| 4 | **移动端适配** | Vue3 响应式改造或独立移动视图 | 开发工作量 |
| 5 | **扫码功能替代** | HTML5 Camera API + jsQR | 开发工作量 |

#### 可选项

| # | 项目 | 说明 |
|:-:|------|------|
| 6 | 通知替代 | 邮件（nodemailer）/ 企业微信机器人 / Web Push |
| 7 | PWA 配置 | manifest.json + Service Worker → 可安装到手机桌面 |
| 8 | Nginx 反向代理 | 前后端同域部署，静态资源缓存 |

### 2.4 纯 Web 方案的优势

| 维度 | 说明 |
|------|------|
| **架构统一** | 一套 Vue3 前端 + 一套 Express 后端 + 一套 MongoDB，彻底消除互通问题 |
| **代码复用** | PC 端已有的全部 32 个 API、8 个 handler、所有页面组件都直接复用，不需要维护两套代码 |
| **MongoDB 全功能** | `$lookup` / `$facet` / `$dateDiff` 全部可用，数据分析扩展无限制 |
| **成本最低** | 一台 ¥40-100/月的轻量服务器搞定全部，不需要腾讯云 MongoDB（¥200-545/月） |
| **开发效率** | 只维护一个项目，前后端本地调试体验统一 |
| **不受平台约束** | 无需微信审核、无需微信开发者工具、不受微信 API 限制 |
| **跨平台** | iOS / Android / PC / 平板，任何有浏览器的设备都能用 |
| **团队分发更简单** | 发个链接即可，不需要微信扫码、不需要关注、不需要体验版名额 |

### 2.5 纯 Web 方案的劣势

| 维度 | 说明 | 严重程度 |
|------|------|:--------:|
| **扫码体验差一些** | HTML5 Camera API 需要用户授权，识别速度略慢于微信原生 `scanCode` | ⚠️ 中 |
| **消息通知弱化** | 失去微信服务通知的强触达能力，替代方案（邮件/企业微信）触达率低一些 | ⚠️ 中 |
| **无微信生态入口** | 不能在微信内分享小程序卡片，需要发链接 | 低 |
| **需自运维服务器** | 需要自己维护服务器安全、备份、更新 | ⚠️ 中 |
| **已有小程序代码废弃** | `miniprogram/` 下 5 个页面 + 6 个云函数 + 订阅消息模块的开发投入沉没 | 高 |
| **手机浏览器体验** | 不如原生小程序流畅（无预加载、下拉刷新需自己实现） | 低 |

## 三、三种方案综合对比

### 3.1 总览

| 维度 | B-1：PC 迁入微信云 | B-2：小程序调外部 API | C：纯 Web（弃小程序） |
|------|:---:|:---:|:---:|
| 保留小程序 | ✅ | ✅ | ❌ |
| 数据互通 | ✅ | ✅ | ✅ |
| MongoDB 全功能 | ❌ 需额外购买 | ✅ | ✅ |
| 架构统一性 | ⚠️ 两套后端 | ✅ 一套后端 | ✅✅ 一套前端+后端 |
| 代码维护量 | 高（两套前端+两套后端） | 中（两套前端+一套后端） | 低（一套前端+一套后端） |
| 平台依赖 | 高 | 中 | 无 |

### 3.2 月度成本对比

| 项目 | B-1 | B-2 | C：纯 Web |
|------|----:|----:|----:|
| 云服务器 / 云托管 | ¥20-50 | ¥40-100 | ¥40-100 |
| 腾讯云 MongoDB | ¥200-545 | ¥0 | ¥0 |
| 微信云开发 | ¥0-19.9 | ¥0 | ¥0 |
| 域名 | ¥5 | ¥5 | ¥5 |
| **月度合计** | **¥225-620** | **¥45-105** | **¥45-105** |
| **年度合计** | **¥2,700-7,440** | **¥540-1,260** | **¥540-1,260** |

### 3.3 开发工作量对比

| 工作项 | B-1 | B-2 | C：纯 Web |
|--------|:---:|:---:|:---:|
| Dockerfile + 容器部署 | 需要 | 不需要 | 不需要 |
| 数据库迁移 | 云 DB → 腾讯云 MongoDB | 保持不变 | 保持不变 |
| 云函数改造 | 6 个全部改造 | 6 个全部改造 | 删除（不再需要） |
| VPC 网络调试 | 需要 | 不需要 | 不需要 |
| 小程序页面改造 | 不需要 | 6 个页面重写调用方式 | 不需要（废弃） |
| 移动端 UI | 不需要（已有） | 不需要（已有） | **需新建**（Vue3 响应式） |
| 扫码功能 | 不需要（已有） | 不需要（已有） | **需新建**（HTML5 Camera） |
| 通知功能 | 不需要（已有） | 不需要（已有） | **需新建**（邮件/企微） |
| 域名备案 | 不需要 | 需要 | 需要 |
| 服务器运维 | 不需要 | 需要 | 需要 |

### 3.4 决策矩阵

| 如果你看重… | 推荐方案 |
|-------------|---------|
| 最低成本 + 最简架构 | **C：纯 Web** |
| 保留微信生态 + 合理成本 | **B-2：小程序调外部 API** |
| 最少改造 + 不计成本 | **B-1：PC 迁入微信云** |
| 长期维护最省心 | **C：纯 Web** |
| 扫码体验最佳 | **B-2：小程序调外部 API** |
| 通知触达率最高 | **B-1 或 B-2**（微信订阅消息） |

---

## 四、结论

对于团队规模小、预算敏感、更看重长期维护成本的场景：

- **首选方案 C（纯 Web）**：架构最简、成本最低、维护最轻，PC 端代码几乎全部复用
- **次选方案 B-2**：保留微信小程序的扫码和通知优势，同时避免高额云数据库费用
- **不推荐方案 B-1**：成本过高（¥200-545/月仅 MongoDB），架构复杂度高
